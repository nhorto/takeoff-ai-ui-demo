# Annotation Deduplication Problem

## The Problem

When the counting agent reads text from architectural stair drawings using `get_page_text()`, it sees **every text annotation** on the page — but construction drawing sheets typically contain **multiple views of the same stair** (plan, section, elevation, axonometric, detail). Each view independently labels the risers and treads, so the same flight's annotation appears multiple times in the extracted text.

The agent has no way to distinguish which annotations belong to which drawing view from text alone. This causes it to **double- or triple-count** flights, producing inflated totals.

### How It Manifests

**Example: Page 258 — Stair 6**

The text dump shows **33 riser/tread annotations** on a single page:

```
12 TREADS @ 11"    (appears 7 times)
13 RISERS @ 6 7/8" (appears 9 times)
9 TREADS @ 11"     (appears 4 times)
9 RISERS @ 6 7/8"  (appears 3 times)
10 RISERS @ 6 7/8"  (appears 3 times)
8 TREADS @ 11"     (appears 2 times)
6 TREADS @ 11"     (appears 1 time)
2 RISERS @ 6 7/8"  (appears 1 time)
7 RISERS @ 6 7/8"  (appears 1 time)
```

The correct answer for Stair 6 is **183 treads, 202 risers** total (across all floors). But if the agent naively sums all 33 annotations, it gets vastly inflated numbers because the same annotations repeat across the section view, plan view, and axonometric view on the sheet.

### Why This Is the #1 Accuracy Bottleneck

Eval scoring shows that tread/riser accuracy is the weakest metric across runs, and the per-stair deltas are consistent (whole-flight overcounts, not individual annotation misreads). The stairs with the most drawing views per page (Stair 2, 3, 5, 6) have the highest error rates. This directly correlates with the annotation duplication.

## Evidence

Generated by `eval/dump-text.ts` — run it to reproduce:

```bash
bun run eval/dump-text.ts --stair 6
```

The output shows exactly what the agent sees via `get_page_text()`, including spatial row positions (Y-coordinates). The duplicate annotations are spatially separated — they're in different regions of the page corresponding to different drawing views.

## Proposed Solutions

### Solution 1: Spatial Clustering

**Approach:** Group annotations by their (x, y) position on the page into spatial clusters that correspond to drawing views. Annotations within the same cluster likely belong to the same view — only count each unique annotation value once per cluster.

**How it works:**
1. Extract all riser/tread annotations with their coordinates
2. Run a spatial clustering algorithm (e.g., DBSCAN or simple grid-based) to identify distinct drawing views
3. Within each cluster, deduplicate annotations with identical values
4. Pick one "canonical" cluster (e.g., the section view, which typically has the most detail) or verify across clusters

**Pros:**
- Purely deterministic — no AI involved, fully reproducible
- Fast — no additional API calls or image processing
- Works with the existing text extraction pipeline
- Can be implemented as a pre-processing step before the agent sees the data
- Spatial data is already available (x, y coordinates from pdf.js)

**Cons:**
- Requires tuning cluster distance thresholds — too tight splits one view, too loose merges views
- Some views overlap spatially (e.g., a detail callout drawn near the section view)
- Doesn't handle cases where different views show *different* flights (e.g., plan shows Level 1-2, section shows Level 2-3) — those aren't true duplicates
- Architecture-specific: threshold values may not transfer well to drawings from different firms or scales

### Solution 2: Text + Image Hybrid Verification

**Approach:** Use text extraction to identify candidate annotations, then use targeted image crops to verify which annotations the agent should count. The image crop lets the agent see the drawing view context.

**How it works:**
1. Text extraction finds all riser/tread annotations with positions
2. For each annotation cluster, crop a small region around it
3. Send the crop to the vision model with a focused prompt: "Is this a plan view, section view, or detail view? Should this annotation be counted?"
4. Agent counts only annotations from the designated view type (e.g., section views only)

**Pros:**
- Most accurate — the model can see the actual drawing context
- Handles edge cases well (overlapping views, unusual layouts)
- Leverages the strength of vision models (spatial understanding)
- Can also catch other issues like misread annotations or unconventional notation

**Cons:**
- Significantly more expensive — each verification crop is an image API call
- Slower — multiple round-trips per page
- Still relies on the model's judgment — could introduce new errors
- More complex implementation (crop coordinates, prompt engineering)
- Cost scales with annotation count: Stair 6 would need ~33 verification crops

### Solution 3: View Title Pre-Labeling

**Approach:** Scan the page text for view titles (e.g., "SECTION - STAIR 6", "STAIR 6 - LEVEL 01 PLAN", "AXONOMETRIC") and associate nearby annotations with those view labels. Then only count annotations from one canonical view type.

**How it works:**
1. Find view title text items (typically larger font, positioned above/below the drawing view)
2. Define a bounding region for each view based on the title position
3. Assign each annotation to the nearest view title
4. Select only one view type for counting (e.g., sections, which label individual flights most explicitly)

**Pros:**
- Deterministic and interpretable — the association logic is transparent
- Cheap — no additional API calls
- Architecturally sound — view titles are a standard convention in construction drawings
- Can provide useful metadata (which view the count came from)

**Cons:**
- View title formats vary between firms ("SECTION", "SECT.", "SEC", "S-", etc.)
- Title position relative to the drawing varies — sometimes above, sometimes below, sometimes in a title strip
- Some pages don't have explicit view titles
- Relies on font size heuristics to identify titles vs. regular annotations
- Multi-page stairs complicate this: Stair 4 spans pages 254+255, where views may span page boundaries

### Solution 4: Hybrid — Spatial Clustering + View Titles (Recommended Starting Point)

**Approach:** Combine Solutions 1 and 3: use view titles to anchor clusters, then use spatial proximity to assign annotations to views.

**How it works:**
1. Find view titles with positions and font sizes
2. Use titles as cluster centroids
3. Assign each annotation to the nearest view title using spatial distance
4. If a page has no view titles, fall back to pure spatial clustering (Solution 1)
5. Count from one canonical view type (section preferred, fall back to plan)

**Pros:**
- More robust than either approach alone
- View titles provide semantic meaning; spatial clustering handles the geometry
- Graceful degradation when titles are missing
- Still fully deterministic, no extra API calls

**Cons:**
- Most complex implementation of the deterministic options
- Still has the edge cases of both approaches
- Needs testing across multiple drawing sets to validate thresholds

## Testing Plan

Each solution can be evaluated within the existing eval framework:

1. **Implement as a pre-processing layer** in `get_page_text()` or as a new tool
2. **Run the standard eval** (`eval/run-eval.ts --runs 3`) against the golden dataset
3. **Compare tread/riser accuracy** before and after using `eval/score-runs.ts --table`
4. **Track cost delta** — Solution 2 will show higher costs, others should be neutral

### Success Criteria
- Stair 6 tread/riser accuracy improves from current ~50-60% to >85%
- Overall tread/riser accuracy across all stairs improves by >15 percentage points
- No regression in stair count accuracy (currently finds 7-8 stairs)
- Cost increase <$0.50 per run (excludes Solution 2 from budget-constrained workflows)
